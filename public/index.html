Geschichte

Geschichte

  • Vorstellung 2009 auf JSConf
  • ab 2012 Verwendung bei Unternehmen (Microsoft, eBay, LinkedIn, The New York Times, Yahoo!, etc.)
  • Ziel: Framework zur einfachen Erstellung von leicht skalierbaren Netzwerkanwendungen
  • über Googles V8 betrieben
  • Eventgesteuerte Programmierung

Vorteile

Vorteile

  • Moderne API, keine Altlasten
  • Geschwindigkeit durch Asynchronizität und Parallelität
  • Cluster/Cloud Applikationen leicht implementierbar
  • Skalierbarkeit: vertikal und horizontal
  • Einfache Handhabung von Multithreading
  • Kein externer Server-Stack (im Gegensatz z.B.: Apache httpd, Apache Tomcat oder nginx)

Vorteile

  • Applikation beinhaltet Server
  • Viele Vendor Bibliotheken (async, moment, express)
  • Schnelle Entwicklung von Applikationen jeder Größe
    • Geringer Konfigurationsaufwand
    • Geringe Fehlerhäufigkeit
    • Konzentration auf Kernproblematik
  • Alle Vorteile eines Applikationsservers
    • Kein Bootstrap pro Request (im Gegensatz zu PHP)
    • Der Bootstrap erfolgt einmalig bei Serverstart

Vorteile

  • Javascript im Front- und Backend (Applikation, Persistenzschicht)
  • Keine Überladenen Frameworks wie PHP’s Zend Framework oder Symfony
  • Native Bindings (C/C++)
  • Mit Node kein Load :)

Nachteile

Nachteile

  • Einführung von neuem Personal gestaltet sich schwierig
  • Noch junger aber aufstrebender Trend (Bleeding Edge)
  • Hohe Qualitätsansprüche an Entwickler, ähnlich Ruby oder Haskell

Erweiterungen, Eigenheiten, Fakten

Erweiterungen, Eigenheiten, Fakten

  • basiert auf Google’s V8 Javascript Engine
    • Incremental Garbage Collector
    • V8 Assembler basiert auf dem Strongtalk Assembler
    • Implementierung von ECMA-262, 5th edition
  • ist zu Teilen in C++, aber größtenteils in Javascript selbst verfasst
    • JavaScript: 70,2%, C++: 23,9%, C: 2,7% Other: 3,1%

Erweiterungen, Eigenheiten, Fakten

  • nutzt libuv, ein abstrakter Platform Layer
  • Dieser ermöglicht den Non-Blocking Event Loop
  • bietet eine REPL (Read, Eval, Print - Loop, ähnlich Ruby oder Python) Umgebung

Dateizugriffe

  • Node erweitert Javascript um die Fähigkeit Dateizugriffe zu realisieren
  • File System Modul bietet (a)synchrone Varianten der Methoden
  • Unterstützung von Streaming (für FS, Sockets, Posix Std-Channels uvm.)

Non-Blocking Event Loop

  • Ist bei Node.js fester Bestandteil des Systems
  • Bietet die Möglichkeit Callbacks für spezifische Events zu feuern
  • Events können sein:
    • POSIX Signals
    • Timeouts und Intervals
    • User-defined Events (EventEmitter)

Non-Blocking Event Loop

  • Vergleichbare Systeme und Bibliotheken sind:
    • libevent für C
    • EventMachine für Ruby
    • vibe.d für D

CommonJS

  • Dient der Standardisierung von Javascript außerhalb eines Browsers (Server-Side Javascript, oder für Desktopanwendungen)
  • Node.js implementiert mehrere der CommonJS Vorschläge (Proposals)
    • Modules, Packages, Filesystem, Binary, Promises

Werkzeuge und Strukturen

Node Package Manager

  • kommt mit Node.js einher
  • organisiert Module so, dass Node.js sie finden kann (node_modules/)
  • intelligente Behandlung von Abhängigkeiten

Node Package Manager

  • npm help - alle verfügbaren Kommandos auflisten
  • npm search - verfügbare Pakete suchen
  • npm install - Pakete installieren
  • npm publish - eigenes Paket zum Index hinzufügen
  • npm ls - installierte Pakete auflisten
  • npm rm - installierte Pakete entfernen

Module

  • Node.js besitzt ein simples Modulsystem
  • package.json beinhaltet Informationen über Modul (Autor, Version, Abhängigkeiten, ...)
  • Core Module sind in die Node.js Binary kompiliert
  • sie werden vorrangig geladen, bspw. lädt require(‘http’) immer das eingebaute HTTP-Modul, auch, wenn ein anderes Modul mit diesem Namen vorhanden ist

Module

  • Dateimodule werden, wenn die Datei nicht gefunden wurde, durch Raten ermittelt
  • beim Raten wird der Dateiname mit den Endungen js, json, und node ergänzt
  • js- und json-Dateien werden als normale Textdateien erkannt, wohingegen node-Dateien als kompilierte Module erkannt werden, die mit dlopen geöffnet werden müssen

Module

  • Modulpfade können sowohl in relativer (./file.js) als auch absoluter (/data/modules/file.js) Schreibweise angegeben werden
  • ohne Angabe von ./ oder / wird angenommen, dass es sich entweder um ein Core Modul, oder eines aus node_modules handelt
  • Export-Beispiel: 
    test = {};
test.firstFunc = function() {...};
module.exports = test;

Praktiken

Genereller Ablauf

  • bad practice (synchronen Code nach Möglichkeit vermeiden) 
    var data = readFromDatabase();
doSomethingWith(data);
doSomeThingUnrelated();
  • good practice (asynchronen Code verwenden) 
    readFromDatabase(function(err, data) {
    doSomethingWith(err, data);
});
doSomeThingUnrelated();

Genereller Ablauf

  • best practice (Events und Ströme verwenden) 
    var query = readFromDatabase();

query.on('err', doFancyErrorHandling);
query.on('data', doSomethingWith);
query.on('end', finalize);

doSomeThingUnrelated();

Callback als Return

  • bad practice 
    if (err) {
   return err;
}
  • good practice 
    if (err) {
   return callback && callback(err);
}

Asynchronous Error Handling

  • bad practice 
    function async(callback) {
    process.nextTick(function(){
           throw new Error("Something went wrong");
           callback && callback(); // Never executed
    });
}

try {
    async(function(){
           console.log("It worked!");
    });
} catch(error) {
    console.log("This is never printed.");
}

Asynchronous Error Handling

  • good practice 
    var emitter = new (require('events').EventEmitter)();

var async = function () {

    process.nextTick(function(){
           emitter.emit('error', new Error('Something went wrong'));
    });

    return emitter;
}

var event = async();
event.on('error', function(error) {
    console.error(error);
});

Synchronous Error Handling

  • good practice 
    var divideBy = function(a,b) {
    if(0 === b) {
           return new Error("Divide by zero!");
    }
    return a / b;
}

var value = divideBy(1,0);
if(value instanceof Error) {
    console.error(value);
};

Adapter

var Vendor = {
    fixedMethod: function(arg1, arg2, arg3) {
        console.log(arguments);
    }
}

Vendor.fixedMethod('defaultValue', {test: 1}, [1, 2, 3]);

Adapter

var Adapter = {
    fixedMethod: function() {

        Vendor.fixedMethod
              .bind(Vendor, 'defaultValue')
              .apply(Vendor, args);
    }
}

Adapter.fixedMethod({test: 1}, [1, 2, 3]);

Über Greppy

Über Greppy

Greppy is a top layer framework utilising the Express framework. Greppy extends the functionality of Express and increases the reuseability for common parts of your application. Furthermore it standardizes an application directory structure which helps you to manage large code bases.

Features

Features

  • Eine wohldefinierte Applikationsstruktur
  • MVC, Kontexte und Module
    • Kontexte umfassen Module, Module enthalten MVC
  • Abstrakte Persistenzschicht
    • Einheitliches Bootstrapping von MySQL und MongoDB Verbindungen
    • Auftrennung von Modellen, Migrationen und Fixtures

Features

  • Leicht zu erweitern und zu konfigurieren
    • Unterstützung von Convention-over-Configuration
    • Einfache Möglichkeiten das Framework zu erweitern
  • Belast- und Skalierbar
    • Cloud/Cluster Software mit Multi-Threading - out of the box
    • Hochverfügbarkeit durch Master/Worker Pattern

Features

  • Rapid Prototyping
    • Generierung von Grundgerüsten, sogar ganze CRUD Controller
    • Schnellere Entwicklung und somit schnellere Resultate

Einsatzbereich

Einsatzbereich

  • Verteilte Anwendungen
  • Hochverfügbare Systeme
  • Umgang mit Big Data
  • Hochperformante System
